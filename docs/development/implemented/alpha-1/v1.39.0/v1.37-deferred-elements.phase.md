## 1. HR Column Mapping Auto-Detection

### Goal

When an admin uploads an HR spreadsheet the column headers are automatically
mapped to the six canonical field kinds (`email`, `name`, `manager`, `position`,
`band`, `unit`) by an AI call, so the column-mapping UI comes pre-filled for
confirmation rather than requiring fully manual entry.

### Why AI, not keyword matching

HR spreadsheets vary widely across organisations. Keyword heuristics break on
abbreviations, non-English headers, or unusual HRIS exports. An LLM call
handles any naming convention without maintaining a keyword list; sample row
values are passed as context to resolve ambiguous headers. The call is a single
`generateObject` with a bounded schema — minimal latency and cost.

### New port

```ts
// packages/domain/src/ports/column-mapping-detector.ts
import type { HrColumnMapping } from "../entities/hr-dataset";
import type { Result } from "../result";

export interface IColumnMappingDetector {
  detect(input: {
    headers: string[];
    sampleRows: Record<string, string>[];  // up to 3 rows for context
  }): Promise<Result<HrColumnMapping>>;
}
```

### Adapter behaviour (`AiColumnMappingDetector`)

- Calls `generateObject` (Vercel AI SDK) with a Zod schema:
  `z.record(z.string(), z.enum(["email","name","manager","position","band","unit"]))`.
- System prompt: given these column headers from an HR spreadsheet, map each one
  to exactly one canonical field kind; omit any header that does not clearly
  belong to a kind.
- Up to 3 sample rows are included in the user prompt as JSON for context.
- On LLM failure: returns `err(domainError("AI_PROVIDER_FAILED", …))`.

### Use-case change (`import-hr-dataset`)

```
if input.columnMapping supplied → use as-is (existing behaviour, unchanged)
else if detector injected      → call detect({ headers, sampleRows: first 3 rows })
     on ok  → store returned mapping
     on err → store empty mapping {}   ← import never fails due to detection
else                           → store empty mapping {}
```

The detector is an **optional** constructor dependency — the use-case stays
fully testable without a real LLM.

A keyword-based `KeywordColumnMappingDetector` stub ships in `packages/adapters`
for unit tests so no test ever hits a real LLM.

## 2. Dynamic Approver Resolution via RAG

### Goal

The `dynamic` approver-source mode currently calls `findPositionHolder` with
only the node's `roleHint` string. This phase adds a RAG step: the policy text
governing the delegation is retrieved from the `kb_` knowledge base, the
relevant role/position is extracted from it, and that extracted role is passed to
`findPositionHolder` — giving the operator a contextually correct suggestion
instead of a plain role-name lookup.

### Approach

No new ports are needed — `IEmbeddingsProvider`, `IDocumentChunkRepository`, and
`ILanguageModel` all exist. The change is entirely within `SuggestApprover`:

1. **Embed** the `roleHint` (plus `instructions` if set) using `IEmbeddingsProvider`.
2. **Search** `IDocumentChunkRepository` scoped to the session's flow for policy
   chunks that describe delegation authority (minSimilarity ≈ 0.75, limit 5).
3. **Extract** the named position/band/unit from the retrieved text with a small
   `generateObject` call (`ILanguageModel`) returning `{ role?, band?, businessUnit? }`.
   If retrieval yields nothing, fall through to the existing `roleHint`-only path.
4. **Call** `resolver.findPositionHolder` with the extracted fields, merging any
   values that the extraction left empty with values from `roleHint`.

The fallback chain is: RAG-extracted fields → `roleHint` → `null`. The
suggestion remains operator-confirmed regardless.

### Key files

| Layer | File | Change |
|-------|------|--------|
| application | `packages/application/src/use-cases/approvals/suggest-approver.ts` | Inject `IEmbeddingsProvider`, `IDocumentChunkRepository`, `ILanguageModel` (all optional); add `resolveWithRag()` private method for `dynamic` path. |
| web | `apps/web/src/lib/container.ts` | Pass existing `embeddings`, `documentChunks`, and `llm` into `SuggestApprover`. |

### Tests

**Unit — `packages/application`** (`suggest-approver.test.ts` additions):
- "dynamic: uses RAG-extracted role to call findPositionHolder when chunks found"
- "dynamic: falls back to roleHint when no chunks retrieved"
- "dynamic: falls back to roleHint when LLM extraction returns empty object"
- RAG dependencies are stubs; no real embedding or LLM call.

**E2E** (extend `phase-step-approvals.spec.ts`):
- A flow with a `dynamic` approval node and a flow-context document containing a
  delegation clause renders a pre-filled suggestion that reflects the policy text.
  (Can be skipped without `TEST_AUTH_BYPASS` like existing approval e2e.)

## 3. Reject / Changes-Requested Route-Back

### Goal

Currently `reject` and `changes_requested` simply hold the session at the
approval node with the comment surfaced. This phase adds routing:

- **Changes requested** → automatically route the session back to the node
  immediately before the approval node, and email the originator the approver's
  comment so they can revise and resubmit.
- **Rejected** → the approver is presented with a binary choice in their inbox
  before confirming the decision: **"Route back to originator"** (same route-back
  as above) or **"Close request"** (cancel the session entirely, notify the
  originator it has been declined with no further action).

### Why no edge schema changes

The "previous node" is already in `session.graphCheckpoint.advancedFrom` (set
by every advance path). Route-back uses that value directly — no new `FlowEdge`
condition field is needed. This keeps the change self-contained to the
application layer.

### Approach

1. **`DecideApprovalInput`** gains an optional `routeBack?: boolean` field (only
   relevant for `rejected`; ignored for `approved` and `changes_requested` which
   always route back).
2. **`decide-approval` non-approve path** (currently returns immediately):
   - Read `session.graphCheckpoint.advancedFrom` for the previous node id.
   - `changes_requested`: always route back — update `session.currentNodeId` to
     the previous node and email the originator.
   - `rejected` + `routeBack: true`: same route-back + email.
   - `rejected` + `routeBack: false` (or missing): set `session.status =
     "cancelled"` and email the originator that the request was declined.
3. **`DecideApprovalOutput`** already has `advanced`, `newNodeId`, and
   `sessionCompleted` — no shape changes needed; `newNodeId` is populated on
   route-back and `sessionCompleted` on cancel.
4. **Approver inbox UI** (`app/(user)/approvals/`) gains a radio/toggle for
   `rejected` decisions: "Route back" vs "Close request". This selection is
   forwarded as `routeBack` to the tRPC `approval.decide` mutation.
5. **`IApprovalDecidedNotifier`** (and its email template) already sends the
   comment; the existing template is updated to note "returned for revision" vs
   "declined" depending on the routing outcome.

### Key files

| Layer | File | Change |
|-------|------|--------|
| application | `packages/application/src/use-cases/approvals/decide-approval.ts` | `routeBack` input field; route-back and cancel paths for non-approve decisions; email originator. |
| application | `packages/application/src/use-cases/notifications/notify-on-approval-decided.ts` | Pass routing outcome to email template ("returned for revision" vs "declined"). |
| web | `apps/web/src/app/(user)/approvals/` | Add "Route back" / "Close request" radio to the reject decision form. |
| web | `apps/web/src/server/routers/approval.ts` | Forward `routeBack` from tRPC input to use-case. |

### Session status for "Close request"

`SessionStatus` already includes (or should include) `"cancelled"`. Verify in
`packages/domain/src/entities/session.ts`; add if absent (no migration — status
is a TS union used as a typed string in JSONB / text column, not a DB enum).

### Tests

**Unit — `packages/application`** (`decide-approval.test.ts` additions):
- "changes_requested: routes session back to previous node and notifies originator"
- "changes_requested: returns advanced=true, newNodeId=previousNodeId"
- "rejected + routeBack: routes session back to previous node"
- "rejected + routeBack:false: cancels session and notifies originator"
- "rejected + no previous node in checkpoint: cancels session (no node to route back to)"

**E2E** (extend `phase-step-approvals.spec.ts`):
- Approver submits changes_requested; session card shows "returned for revision" + comment.
- Approver submits rejected with "Close request"; session card shows "declined".

- [ ] E2E: upload CSV → mapping UI pre-filled.

- [ ] E2E: `dynamic` node with a policy document yields a contextual suggestion.
- [ ] `decide-approval` with `changes_requested`: routes session back to previous node; originator emailed.
- [ ] `decide-approval` with `rejected` + `routeBack: true`: same route-back behaviour.
- [ ] `decide-approval` with `rejected` + `routeBack: false`: session cancelled; originator emailed "declined".
- [ ] Approver inbox UI shows "Route back" / "Close request" choice on reject.
- [ ] `./validate.sh` passes; `VERSION` = `1.38.0`.

## 5. Out of scope

- Auto-invite / provisioning of free-typed approvers (deferred indefinitely; current email + sign-up flow is acceptable).
- Confidence scores or per-field uncertainty on detected column mappings.
- Re-running detection on an already-mapped dataset.
- Multi-hop route-back (more than one node back).
- Delegation / quorum / parallel approval chains.
