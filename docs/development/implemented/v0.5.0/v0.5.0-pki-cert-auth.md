# Phase v0.5.0 ŌĆö PKI Certificate Authentication

## Summary

Add PKI/client-certificate authentication as a selectable auth method alongside the existing magic link flow. The reverse proxy (nginx/Caddy) performs mTLS termination and forwards certificate identity via request headers; the app reads those headers, provisions users on first login (JIT), and creates a Better Auth session. The `init-project.sh` script gains an auth-method selection step so new projects can choose their auth strategy at bootstrap time.

## Why

Enterprise and government deployments commonly issue certificates to user workstations via a certificate management app. Users expect to log in by presenting that cert ŌĆö no email flow, no password. This change makes the template fit for those environments without removing magic link for teams that don't use PKI.

## Architecture

### Header contract (reverse proxy ŌåÆ app)

| Header | Content | Required |
|---|---|---|
| `X-SSL-Client-Verified` | `SUCCESS` or `FAILED` | Yes |
| `X-SSL-Client-Subject-DN` | Full Subject DN, e.g. `CN=Jane Smith,OU=Eng,O=Acme` | Yes |
| `X-SSL-Client-Fingerprint` | SHA-256 fingerprint, e.g. `sha256:abc123...` | Yes |
| `X-SSL-Client-SAN-Email` | Email from Subject Alt Name (preferred) | No |

### Identity extraction

1. Use `X-SSL-Client-SAN-Email` if present and non-empty ŌåÆ email
2. Fall back to parsing `CN=<value>` from Subject DN and treating it as display name; require SAN email or reject

### Trusted proxy protection

Cert headers MUST be stripped from any request not originating from a trusted proxy IP. `PKI_TRUSTED_PROXY_IPS` is validated on startup; the cert route returns 400 if the source IP is not in the list. This prevents a direct client from spoofing cert headers.

### Session creation

`PkiCertAdapter` inserts directly into `core_sessions` (same pattern as the `dev-login` route). Better Auth's own session table is used so `resolveSession` works unchanged. Session TTL: 8 hours (configurable via `PKI_SESSION_TTL_HOURS`, default 8).

### User provisioning (JIT)

1. Look up user by email via `IUserRepository.findByEmail`
2. If not found: create with name from CN, email from SAN, `isAdmin: false`
3. Update `cert_fingerprint` and `cert_subject_dn` on every login (handles cert rotation)

### Auth method configuration

`createAuth` accepts a new `authMethod` discriminated union:

```typescript
type AuthMethod =
  | { type: "magic-link"; sendMagicLink: (p: { email: string; url: string }) => Promise<void> }
  | { type: "pki"; pkiConfig: PkiConfig }
  | { type: "pki-and-magic-link"; pkiConfig: PkiConfig; sendMagicLink: ... }
  | { type: "google-oauth" }   // scaffold ŌĆö throws at runtime with setup instructions
  | { type: "other" }          // no-op placeholder
```

The magic link plugin is only added when `authMethod.type` includes `magic-link`.

## Sub-components (build order)

### 1. DB schema + migration
- Add `cert_fingerprint text` and `cert_subject_dn text` (both nullable) to `core_users`
- Write and run Drizzle migration

### 2. `PkiCertAdapter` (packages/adapters/src/auth/pki-cert-adapter.ts)
- Validates trusted proxy source
- Parses headers ŌåÆ `CertIdentity { email, name, fingerprint, subjectDn }`
- JIT-provisions user via `IUserRepository`
- Inserts session into `core_sessions`
- Returns `Result<{ token: string; userId: string }>` ŌĆö never throws

### 3. Updated `createAuth` factory
- Add `authMethod` discriminant to `AuthConfig`
- Conditionally include `magicLink` plugin
- Export `AuthMethod` type

### 4. PKI cert route (`apps/web/src/app/api/auth/cert/route.ts`)
- `POST /api/auth/cert?redirect=<path>`
- Calls `PkiCertAdapter.authenticate(request.headers, sourceIp)`
- On success: sets `better-auth.session_token` cookie, redirects to `redirect` param (validated to be a relative path)
- On failure: returns 401 JSON

### 5. Updated middleware
- When `AUTH_METHOD=pki` or `AUTH_METHOD=pki-and-magic-link`: redirect unauthenticated requests to `/api/auth/cert?redirect=<original>`
- When `AUTH_METHOD=magic-link` (default): existing behaviour unchanged

### 6. Init script auth selection
- Insert prompt between AI provider and Langfuse steps
- Write chosen `AUTH_METHOD=<value>` into `.env.example`
- If PKI selected: add `PKI_TRUSTED_PROXY_IPS=` placeholder and instructions
- If Google OAuth selected: add warning and placeholder keys; warn "requires additional setup ŌĆö see docs/guides/google-oauth.md"

## Acceptance Criteria

All items must pass before the phase is considered complete.

- [ ] A request arriving at `/api/auth/cert` from a trusted proxy IP with `X-SSL-Client-Verified: SUCCESS`, a valid Subject DN, and a SAN email receives a `Set-Cookie: better-auth.session_token=...` header and a `302` redirect to the `redirect` query param
- [ ] A request from an IP not in `PKI_TRUSTED_PROXY_IPS` receives `400` regardless of cert headers present
- [ ] A request with `X-SSL-Client-Verified: FAILED` receives `401`
- [ ] A request with no SAN email and no parseable CN email receives `401`
- [ ] First login for an unknown email creates a new user row in `core_users` with `is_admin = false`
- [ ] Subsequent login for a known email updates `cert_fingerprint` and `cert_subject_dn` without creating a duplicate user
- [ ] `?redirect=/../../etc` (path traversal) and `?redirect=https://evil.com` (absolute URL) are rejected; redirect defaults to `/`
- [ ] When `AUTH_METHOD=magic-link`, middleware behaviour is unchanged from pre-0.5.0
- [ ] When `AUTH_METHOD=pki`, unauthenticated requests to `/admin/*` redirect to `/api/auth/cert?redirect=<original>` rather than `/admin/login`
- [ ] `init-project.sh` presents the auth method menu and writes the chosen `AUTH_METHOD` to `.env.example`
- [ ] `./validate.sh` passes with zero errors after all changes

## Risks

| Risk | Likelihood | Severity | Mitigation |
|---|---|---|---|
| `PKI_TRUSTED_PROXY_IPS` left empty or misconfigured | Medium | Critical | Startup validation: if `AUTH_METHOD` includes `pki` and `PKI_TRUSTED_PROXY_IPS` is empty, the adapter throws at construction time rather than failing open at runtime |
| JIT provisioning grants access to any holder of a CA-signed cert | Low | High | Documented explicitly ŌĆö deployers must control which CA(s) the reverse proxy trusts; the app trusts the proxy's validation. If an allowlist is needed it is a future enhancement |
| Open redirect via `?redirect=` parameter | Medium | Medium | Validate that redirect value is a relative path (starts with `/`, no `//`, no protocol) before redirecting; default to `/` on invalid input |
| Cert headers injected by a compromised internal service | Low | High | Source IP check covers this for network-level compromise; host-level compromise is out of scope |
| Session cookie not marked `HttpOnly`/`Secure` | Low | High | Inherit Better Auth's cookie defaults ŌĆö verify in tests that the cookie flags match the existing magic link session cookie |
